# 사업시도 1차 — 프로젝트 헌법

## 프로젝트 목적

홈페이지가 없는 로컬 업체를 발굴하고, Claude로 웹사이트를 제작해 판매하는 수익화 실험.
**목표**: 첫 유료 클라이언트 확보 → 반복 가능한 제작·판매 시스템 구축

## 비즈니스 맥락

- 타겟: 단가 있고 예약·상담이 중요한 지역 업종 (예: 네일샵, 피부관리실, 헤어샵, 학원, 맛집 등)
- 조건: 리뷰 많고 활발히 영업 중 + 홈페이지 없음
- 제작 도구: Claude 웹 (claude.ai) + getdesign.md 와이어프레임
- 호스팅: 아임웹 (HTML 내보내기 연결)
- 영업 방식: 먼저 결과물 제시 → "브랜드 보고 만들어봤습니다" → 무부담 제안

## 수익 구조

| 단계 | 항목 | 예상 단가 |
|------|------|----------|
| 초기 | 웹사이트 제작비 | 30~100만 원 |
| 지속 | 유지보수·호스팅 관리 | 월 3~10만 원 |
| 확장 | SEO·블로그·예약 시스템 | 별도 견적 |

---

## ⚠️ 핵심 원칙: 5단계 강제 분리

```
Research → Plan → Execute → Review → Ship
```

**한 턴에 다 섞으면 품질이 망한다. 이것이 이 시스템의 존재 이유다.**

- 각 단계는 역할·담당 모델·산출물이 다르다
- 다음 단계는 이전 단계 산출물만 근거로 진행한다
- 사람(나)은 각 단계 후 승인자 역할만 한다
- 승인 없이 다음 단계로 넘어가지 않는다

---

## 단계별 담당 및 규칙

| 단계 | 담당 | 산출물 | 핵심 금지 |
|------|------|--------|-----------|
| Research | 제한 없음 | RESEARCH.md | 코드·구현·UI 생성 |
| Plan | **Claude Opus** | PLAN.md | 구현·코드 작성 |
| Execute | **Claude Sonnet** | 실제 코드 | 설계 변경·기능 확장 |
| Review | **QA 에이전트** | REVIEW.md | 기능 추가·범위 확장 |
| Ship | 제한 없음 | PR + 배포 패키지 | 기능 추가 |

### Research 규칙

- 코드·구현·UI 생성 금지
- 행동형 검색 사용 ("~하는 법", "~찾는 법")
- 코드베이스 먼저 읽고 도메인 지식 수집
- 산출물: `RESEARCH.md`
- "무엇을 만들까"보다 "무엇을 만들면 안 되는가" 먼저 정의

### Plan 규칙 (Opus 담당)

- 구현·코드 작성 금지
- 옵션 3개 도출 + 트레이드오프 분석
- 1개 채택 + 버린 이유 기록
- "가장 멋진 구조"보다 "오늘 배포 가능한 가장 단순한 구조"
- 설계가 끝난 후에만 Execute 허용
- 산출물: `PLAN.md`

### Execute 규칙 (Sonnet 담당)

- PLAN.md 밖 기능 추가 금지
- 설계 변경·기능 확장 금지
- 구현 중 새 아이디어 → 보고만, 구현 금지
- 항상 빌드 가능한 상태 유지

### Review 규칙 (QA 에이전트)

- 기능 추가·범위 확장 금지
- 구현자 관점 아닌 사용자·클라이언트 관점
- 테스트 + 엣지케이스 + 리스크 점검
- 사람은 최종 평가자 역할만
- 산출물: `REVIEW.md`

### Ship 규칙

- 기능 추가 금지
- Git 커밋 + PR 생성 + 배포 자동 처리
- 클라이언트 전화 스크립트 준비
- 남은 한계와 추후 개선 후보는 기록만

## 사람 승인 게이트

| 게이트 | 조건 | 승인 방법 |
|--------|------|-----------|
| Research → Plan | RESEARCH.md 검토 후 | "승인" 또는 "Plan 시작" |
| Plan → Execute | PLAN.md 검토 후 | "승인" 또는 "Execute 시작" |
| Execute → Review | 구현 결과 확인 후 | "승인" 또는 "Review 시작" |
| Review → Ship | REVIEW.md 검토 후 | "승인" 또는 "Ship 시작" |
| Ship → 배포 | PR 확인 후 | "배포 승인" |

---

## 금지 사항 (절대 타협 없음)

- 단계 건너뛰기
- Research/Plan 단계에서 코드 작성
- Execute 단계에서 설계 변경
- Review/Ship 단계에서 기능 추가
- 승인 없이 다음 단계 진행
- 한 턴에 여러 단계 혼합
